# Skill: sec

Секреты хранятся в XChaCha20-Poly1305-файле, мастер-ключ — в macOS Keychain.
`sec` — отдельный CLI; ставится через Homebrew, исходники — в репозитории
github.com/kaidstor/sec.

## Установка

CLI ставится готовым бинарём через Homebrew (Go не нужен):

```
brew install kaidstor/tap/sec
sec version
```

Дальше зови просто `sec ...` (бинарь в `PATH`). Если команда `sec` не находится —
значит не установлена: поставь строкой выше. В установленный скилл (`npx skills
add kaidstor/sec`) входит только этот `SKILL.md` — сам бинарь всегда из brew,
исходники в скилл не копируются.

Сборка из исходников (для разработки самого CLI) и выпуск новых версий в
Homebrew (тег `vX.Y.Z`) — в репозитории github.com/kaidstor/sec (`cli/README.md`).

## Адресация

Секрет = `<proj>/<KEY>` (например `whois/API_TOKEN`). Если проект не указан,
берётся имя текущей директории — внутри папки проекта достаточно `sec
run -- just dev` или `sec set API_TOKEN`.

**Инстансы/окружения (`-e/--env`).** У сервиса может быть несколько наборов
значений одних и тех же ключей — разные компании/стенды (`commercial`, `max`,
`prod`, …). Это необязательная третья ось: `<proj>` = сервис, `-e <инстанс>`
выбирает набор. Внутри стор это отдельный проект `service@env`, поэтому сервисы
без инстансов работают как раньше. Флаг `-e` действует почти во всех командах
над значениями:

```sh
sec set some-bot/BOT_TOKEN -e commercial   # значение инстанса commercial
sec get some-bot/BOT_TOKEN -e max          # значение инстанса max
sec run some-bot -e commercial -- just start
sec ls some-bot                            # покажет инстансы (commercial, max)
sec diff some-bot -e commercial max        # сравнить инстансы, без раскрытия
```

Конвенция для агента: **сервис = проект, инстанс через `-e`, объявляется в
`.sec`**. Файл `.sec` в папке сервиса декларирует набор инстансов и дефолт, его
читают и CLI (подставляет дефолт, когда `-e` не задан), и агент:

```
project: some-bot
envs:    commercial, max
default: commercial
keys:    BOT_TOKEN, DB_PASSWORD, COMPANY_ID
```

Тогда в папке `sec run -- just start` возьмёт дефолтный инстанс, `-e max`
переопределит, а `sec check`/`sec check --all-envs` проверят, что у инстанса(ов)
заведены все ключи. У `import`/`push` `-e` задаёт инстанс в sec, а окружение
Infisical — отдельный `--infisical-env` (по умолчанию = значению `-e`).

## Правила для агента (главное)

1. **Никогда не выводить значения секретов в чат.** `get` без `--clip` —
   только если пользователь явно попросил показать значение.
2. **Потребление секретов** — всегда без показа значения:
   - запустить процесс: `sec run <proj> -- cmd args...`
     (env-инъекция; `--only A,B` — только выбранные ключи; `-v` покажет
     имена ключей в stderr — имена безопасны);
   - нужен .env: `sec export <proj> --file .env` (в stdout export
     не пишет принципиально);
   - проверить наличие: `sec ls [proj]` — там только имена, можно свободно;
   - **искать нужный ключ — фильтром, а не глазами по всему списку:**
     `sec find <шаблон>` ищет по всему хранилищу и печатает адреса
     `proj/KEY` (шаблон — подстрока без учёта регистра либо glob `*`/`?`;
     со слэшем внутри левая часть матчится на проект: `sec find 'gidcaf/*TOKEN'`).
     Ничего не нашлось — код возврата 1. Сузить конкретный список —
     `sec ls [proj] --filter <шаблон>` (фильтрует имена проектов, инстансов
     или ключей — того, что команда сейчас показывает). Не выгребай
     `sec ls`/`sec ls --json` целиком, чтобы искать в выводе руками;
   - свериться, *какое именно* значение лежит, не раскрывая его:
     `sec get <proj>/<KEY> --peek` (маска `ab…yz` + длина),
     `sec get <proj>/<KEY> --fingerprint` (отпечаток `fp:…`, безопасен для
     чата — по нему нельзя перебрать значение без мастер-ключа) или
     `sec history <proj>/<KEY>` (то же по всем версиям);
   - проверить, совпадает ли значение с тем, что у пользователя/задеплоено,
     не раскрывая ни то ни другое: `sec verify <proj>/<KEY>` (кандидат из
     скрытого ввода/stdin/буфера, ответ «совпадает/нет»);
   - сравнить два проекта (staging vs prod и т.п.) по отпечаткам:
     `sec diff <projA> <projB>` — какие ключи совпадают/различаются;
   - 2FA-код: `sec otp <proj>/<KEY>` — печатать код в чат безопасно:
     TOTP живёт ≤30 секунд, HOTP (seed `otpauth://hotp/…&counter=N`)
     одноразовый — команда сама сдвигает счётчик в сторе;
   - файловый секрет (сертификат, ключ) нужен на диске:
     `sec get <proj>/<KEY> --out <путь>` (файл 0600; бинарные значения
     по-другому не достаются — в чат/буфер/env они не отдаются);
   - конфиг не-env формата: `sec render config.tpl --file config.yml` —
     шаблон с `{{ secret "proj/KEY" }}`, вывод только в файл;
   - машинный вывод для скриптов: `sec ls --json`, `sec info --json`
     (значений там нет — только имена/метаданные).
3. **Заведение секрета** — значение не должно пройти через чат. Новый
   пароль/токен, который нужно только придумать, — генерируй сам, не видя
   значения: `sec gen <proj>/<KEY> --len 32 [--symbols] [--clip]` (`--clip`
   положит в буфер, чтобы пользователь вставил в форму; на macOS запись
   помечается concealed — менеджеры истории буфера типа Raycast её не
   сохраняют). Существующее значение — два пути:
   - **буфер обмена (основной для агентной сессии):** попросить пользователя
     скопировать значение в буфер (Cmd+C), затем выполнить
     `sec set <proj>/<KEY> --clipboard --clear` — значение сохранится,
     агент его не увидит, буфер очистится. Копия пользователя могла
     осесть в истории менеджера буфера (Raycast и т.п.) — если у него
     такой стоит, напомнить почистить историю;
   - **скрытый ввод (у пользователя есть терминал):** предложить выполнить
     самому `sec set <proj>/<KEY>` — CLI дважды спросит значение с
     выключенным echo.
   - **секрет — это файл целиком** (сертификат `.p12`/`.pem`, приватный
     ключ, keystore): `sec set <proj>/<KEY> --from-file <путь>` — текст
     хранится байт-в-байт, бинарь — как base64 (лимит 4 МиБ); обратно —
     только `sec get <proj>/<KEY> --out <путь>`; исходный файл после
     проверки удалить;
   - **значение уже лежит открытым файлом на диске** (`~/.config/<tool>/token`,
     сиротский `.env` и т.п.): не потреблять его паттерном
     `export KEY=$(cat <файл>)` в каждой команде — один раз мигрировать в стор
     пайпом `printf '%s' "$(cat <файл>)" | sec set <proj>/<KEY>` (значение не
     проходит через argv/чат), потреблять дальше через
     `sec run <proj> --only <KEY> -- cmd ...`, и после проверки, что
     потребление работает, удалить исходный plaintext-файл.
   - Если пользователь уже прислал значение открытым текстом в чат — можно
     сохранить через `printf '%s' '<значение>' | sec set <proj>/<KEY>`,
     но предупредить, что оно засвечено в переписке, и предложить ротацию.
4. **Миграция существующих .env / JSON:** `sec import [proj] path/to/.env`
   (путь позиционно или флагом `--file`, умолчание — `./.env`; проект —
   аргументом или по имени текущей папки). Импортирует все KEY=VALUE,
   значения не печатает; `#` внутри значения, кавычки, краевые пробелы,
   BOM и CRLF обрабатываются, битые строки — предупреждением в stderr.
   После — предложить пользователю удалить исходный .env.
   **JSON** — тот же `import`, формат определяется по содержимому:
   `cat creds.json | sec import <proj>` (пайп), `sec import <proj> creds.json`
   (файл), `sec import <proj> --clipboard` (буфер). Вложенные объекты
   разворачиваются через `_` (`{"db":{"pass":…}}` → `DB_PASS`), ключи
   приводятся к имени env-переменной. Форма `sec import <proj> '{"KEY":"…"}'`
   тоже работает, но **значения уходят в argv и историю shell** — агенту
   так не делать: пайп или файл; если JSON уже засветился в чате,
   предупредить и предложить ротацию. Когда JSON нужен целиком одним
   секретом (service-account ключ) — это не import, а
   `sec set <proj>/<KEY> < creds.json`. Источником может быть и **Infisical**
   (через их CLI, нужен `infisical login`): `sec import <proj>
   [-e инстанс] --from-infisical [--infisical-env dev --path / --projectId <id>]`
   — значения идут из stdout экспорта, не через argv. Обратно, из локального
   стора в Infisical: `sec push <proj> [-e инстанс] --to-infisical [--infisical-env … --path …]`
   (значения уходят временным .env-файлом 0600, не через argv). Роль:
   sec — локальный агент-безопасный черновик, Infisical — командный
   источник истины; без `--projectId` проект берётся из `.infisical.json`
   в текущей папке (как `infisical run/export`).
5. Перед `rm` (особенно `rm <proj> --all`) — подтверждение пользователя.
   Если на ключ/проект кто-то ссылается или наследует, `rm` (и `get --once`)
   печатает предупреждение — какие ссылки станут битыми и кто потеряет
   унаследованные ключи (удаление всё равно выполняется). Перезапись через
   `set`/`import`/`gen` безопаснее: старое значение уходит в историю
   (до 5 версий). Навигация по истории — пара undo/redo (как в
   редакторе): `sec undo <proj>/<KEY>` шагает на версию назад (повторный —
   ещё глубже, до стены), `sec redo <proj>/<KEY>` возвращает вперёд; новый
   `set` обнуляет redo. `sec get <proj>/<KEY> --prev 1` читает старое, не
   меняя текущего (правила показа — как у get). Реорганизация — через
   `sec mv <proj>/<KEY> <p2>[/<KEY2>]`, а не get+set: mv не раскрывает
   значение и сохраняет историю.
   **Ротация скомпрометированного секрета:** после `gen`/`set` нового
   значения старое (утёкшее) остаётся в истории до 5 перезаписей —
   `sec forget <proj>/<KEY>` вычищает историю и redo, оставляя только
   текущее значение. Делать это после того, как убедились, что новое
   значение рабочее (forget необратим, `undo` его уже не вернёт).
6. **Журнал:** все обращения к секретам логируются (имена, операции, кто —
   без значений). `sec log [proj[/KEY]]` — показать пользователю, кто и
   когда трогал секреты (`--json` — машинно).
7. **Не дать секрету утечь наружу.** Перед тем как коммитить/публиковать
   файлы или писать значение в файл, прогоняй `sec scan`:
   - `sec scan <путь...>` — ищет сохранённые значения в файлах/каталогах;
   - `sec scan --staged` — проверяет `git diff --cached` перед коммитом
     (годится как pre-commit хук: ненулевой код блокирует коммит);
   - `sec scan -` — проверить произвольный текст из stdin.
   Печатает только `файл:строка → ключ`, значение не раскрывает. Нашлось —
   предупреди пользователя и предложи убрать значение / заменить на
   `sec run`/`sec export`.
   **Показать лог/дифф/вывод команды, в котором может быть секрет** — сначала
   прогони через `sec redact` (обратная сторона scan: не находит, а вычищает):
   `<команда> 2>&1 | sec redact` или `sec redact <файл>` — заменяет каждое
   сохранённое значение на `[redacted:proj/KEY]` и отдаёт безопасный текст в
   stdout (его уже можно показывать). Так вывод команды с секретом в env/аргументах
   не утащит значение в чат. `--mask` — глухое `[redacted]` без имени ключа,
   `--history` — чистить и прошлые значения. Redact видит только значения из
   стора (как scan) — не универсальный DLP.
8. **Гигиена ротации.** Несекретные метаданные ключа (назначение, где и как
   часто крутить) — `sec meta <proj>/<KEY> [--note … --kind … --rotate-url …
   --rotate-every 90d | --expires 2026-12-01]`; их же удобно проставлять при
   заведении: `sec set/gen … --note … --kind …`. Что пора ротировать —
   `sec stale` (ненулевой код — гейт для CI), общее здоровье стора (права,
   дубли значений, просрочки) — `sec doctor`. Требуемый набор ключей проекта
   можно задать манифестом `.sec` и проверять `sec check`.
9. **Общие значения между проектами/инстансами — связывай, не копируй.** Если
   одно значение нужно нескольким пачкам, единый источник правды держится
   ссылкой/наследованием (правка в родителе видна везде сразу, не расходится):
   - одно значение: `sec link <proj>/<KEY> <род>/<KEY2>` — `proj/KEY` отдаёт
     живое значение родителя (инстанс родителя — `--parent-env`);
   - вся пачка: `sec extend <proj> --from <род>` — `proj` видит ключи родителя
     read-only, свои перекрывают; `run`/`export` инъектят объединение.
   Значение по ссылке/наследованию **read-only в потомке**: `set`/`gen`/`undo`/
   `import` откажутся и подскажут родителя. Менять — у родителя (`sec set
   <род>/<KEY>`); сделать собственным — `sec unlink <proj>/<KEY>`; дать своё
   поверх унаследованного — `sec set <proj>/<KEY> --override`. В `sec ls <proj>`
   ссылки помечены `→`, унаследованные `⤷`.

## Типовые сценарии

```sh
# найти, где лежит нужный ключ, не листая хранилище целиком
sec find token             # все проекты/ключи, где встречается "token"
sec find '*_URL' -l        # glob + даты и метаданные
sec find 'gidcaf/*TOKEN'   # только в проекте gidcaf
sec ls --filter git        # сузить список проектов
sec ls gidcaf --filter TOKEN   # сузить список ключей проекта

# запустить дев-сервер с секретами проекта (внутри его папки)
sec run -- just dev

# скрипту нужен один токен
sec run findler --only YT_TOKEN -- ./sync.sh

# завести новый секрет, не увидев его (пользователь скопировал в буфер)
sec set whois/REGISTRY_KEY --clipboard --clear

# затащить существующий .env в хранилище и собрать его обратно на другой машине
sec import whois ~/Projects/whois/.env   # путь позиционно; проект — из аргумента
sec import .env.prod                     # проект = имя текущей папки
cat creds.json | sec import whois        # JSON пайпом (формат определится сам)
sec export whois --file .env

# токен перезаписали, нужен старый
sec history whois/API_TOKEN     # свериться по маске/дате (видно и redo-версии)
sec undo whois/API_TOKEN        # шаг назад (sec redo — вперёд)

# ротация: сгенерировали новый, старый утёкший убираем из истории
sec gen whois/API_TOKEN --len 40
sec forget whois/API_TOKEN      # когда новый проверен рабочим

# пройти 2FA: seed уже сохранён обычным set
sec otp github/OTP

# показать, кто трогал секреты проекта
sec log whois

# придумать новый сильный пароль для регистрации, не увидев его
sec gen whois/ADMIN_PASS --len 24 --symbols --clip

# перенести секреты на другую машину / сервер (passphrase, Keychain не нужен)
sec backup --file sec-bak.enc          # там: sec restore --file sec-bak.enc

# конфиг не-env формата из шаблона
sec render config.tpl --file config.yml

# проверить, что перед коммитом никакой секрет не утёк в индекс
sec scan --staged

# безопасно показать вывод команды, в котором может быть секрет
just dev 2>&1 | sec redact            # значения → [redacted:proj/KEY]

# завести токен с описанием и политикой ротации
sec set whois/API_TOKEN --kind apikey --note "токен реестра"
sec meta whois/API_TOKEN --rotate-every 90d --rotate-url https://dash/keys
sec stale            # что уже пора крутить

# сверить, что задеплоенный токен совпадает с сохранённым (без раскрытия)
sec verify whois/API_TOKEN            # вставить/подать кандидата
sec diff staging prod                 # где значения разошлись

# общее значение для нескольких сервисов — одна ссылка, единый источник правды
sec link findler/DATABASE_URL shared/DATABASE_URL # findler берёт значение из shared
sec extend findler --from shared                  # findler видит все ключи shared read-only

# одноразовая передача значения и авто-очистка буфера
sec get whois/API_TOKEN --once                    # показать и удалить
sec get whois/ADMIN_PASS --clip --clear-after 20s # в буфер, сотрётся через 20с

# синхронизация между машинами через общий зашифрованный блоб
sec sync --file ~/Dropbox/sec.enc     # merge из блоба + push обратно

# обмен с Infisical (нужен infisical login; проект — из .infisical.json)
sec import whois --from-infisical --infisical-env dev   # Infisical → локальный стор
sec push whois --to-infisical --infisical-env dev       # локальный стор → Infisical

# скомпрометирован мастер-ключ — ротировать и перешифровать стор
sec rekey
```

## Не сработает

- Нет TTY и нечего взять из буфера/stdin — `set` со скрытым вводом требует
  терминал; в headless-сценарии подавай значение пайпом.
- На Linux мастер-ключ идёт в Secret Service / GNOME Keyring через
  `secret-tool` (libsecret), если он установлен и поднят; иначе — файл
  `~/.config/sec/key` (0600) либо `SEC_KEY`. На headless-сервере штатный
  путь — `SEC_KEY`/файл. Между машинами (mac↔Linux) синхронизация через
  `sec sync --file` работает независимо от бэкенда ключа (блоб под passphrase).

sec.

локальный менеджер секретов для агентных сессий

Хранилище — один файл XChaCha20‑Poly1305, мастер‑ключ в macOS Keychain. Сделан так, чтобы значения секретов не попадали в argv, историю shell и чат агента: ввод скрытый / из stdin / из буфера, потребление — через env‑инъекцию или запись в файл.

🔒 Значение секрета не проходит через argv и не печатается в чат — если только вы явно не попросили.

Быстрый старт

CLI поставляется со скиллом; для сборки нужен только Go (единственная зависимость — golang.org/x/crypto для Argon2id — завендорена, сборка офлайн). Wrapper собирает бинарь при первом вызове:

"<skill-dir>/sec.sh" ls  ·  либо cd cli && just build → ../bin/sec  ·  just install → прямой вызов sec

Адрес секрета — <proj>/<KEY>. Если проект опущен, берётся имя текущей папки: внутри ~/Projects/rebrandy/whois достаточно sec run -- just dev.

Завести секрет

set — сохранить

sec set whois/API_TOKEN
скрытый ввод дважды (нужен TTY)
sec set whois/API_TOKEN --clipboard --clear
взять из буфера и очистить его
cat token.txt | sec set whois/API_TOKEN
из stdin / файла
sec set whois/API_TOKEN --note "реестр" --kind apikey
+ метаданные сразу

gen — придумать, не видя

sec gen whois/ADMIN --len 24 --symbols
криптостойкое значение, агент его не видит
sec gen whois/ADMIN --len 32 --clip
+ положить в буфер для вставки в форму

Получить и потребить

get — прочитать

sec get whois/API_TOKEN
напечатать (только по явной просьбе)
sec get whois/API_TOKEN --clip --clear-after 20s
молча в буфер, авто‑очистка через 20с
sec get whois/API_TOKEN --once
показать и сразу удалить ключ
sec get whois/API_TOKEN --prev 1
предыдущее значение из истории

run / export / render

sec run whois -- just dev
env‑инъекция секретов проекта в процесс
sec run findler --only YT_TOKEN -- ./sync.sh
только выбранные ключи
sec export whois --file .env
собрать .env (только в файл, 0600)
sec render config.tpl --file config.yml
шаблон {{ secret "proj/KEY" }} → файл

Свериться, не раскрывая значение

маска и отпечаток

безопасно для чата
sec get whois/API_TOKEN --peek
маска ab…yz + длина
sec get whois/API_TOKEN --fingerprint
fp:… — HMAC на мастер‑ключе, не перебрать

verify / diff

sec verify whois/API_TOKEN
совпадает ли кандидат с сохранённым (да/нет)
sec diff staging prod
какие ключи совпадают/различаются

История: undo / redo / forget

Навигация как в редакторе

Каждый set поверх ключа утаскивает старое значение в историю (до 5 версий). undo шагает на версию назад (повторный — глубже, до самой старой, потом стена), redo возвращает вперёд; новый set обнуляет redo.

sec history whois/API_TOKEN [--json]
sec undo whois/API_TOKEN · sec redo whois/API_TOKEN

Ротация скомпрометированного

Обычный gen/set не удаляет старое значение — оно живёт в истории ещё до 5 перезаписей. forget вычищает историю и redo, оставляя только текущее (необратимо — после проверки нового).

sec gen whois/API_TOKEN --len 40
sec forget whois/API_TOKEN

Метаданные и гигиена ротации

meta — назначение и политика

sec meta whois/API_TOKEN --note "реестр" --kind apikey
sec meta whois/API_TOKEN --rotate-every 90d --rotate-url https://dash/keys
sec meta whois/API_TOKEN --expires 2026-12-01

stale / doctor

sec stale [--older-than 90d]
что пора ротировать (ненулевой код — гейт CI)
sec doctor
права стора, бэкенд ключа, дубли значений, просрочки

Защита от утечки

scan — найти сохранённые значения в открытом виде

Ловит секрет до коммита/лога/публикации. Печатает только файл:строка → ключ, значение не раскрывает; ненулевой код при находках — годится как pre‑commit хук / CI‑гейт.

sec scan src/ .env config/
файлы и каталоги
sec scan --staged
git diff --cached (перед коммитом)

redact — вычистить значения из текста

Обратная сторона scan: не находит, а отдаёт очищенный текст — каждое сохранённое значение меняет на [redacted:proj/KEY]. Безопасно показать лог/дифф/вывод команды в чате.

just dev 2>&1 | sec redact
очистить вывод команды → stdout
sec redact app.log --file safe.log
результат в файл 0600

Инстансы / окружения

Один сервис — несколько наборов

Разные компании/стенды (commercial, max, prod) — одни ключи, свои значения. Необязательная третья ось: <proj> — сервис, -e <инстанс> выбирает набор. Внутри — проект service@env; сервисы без инстансов не меняются.

sec set some-bot/BOT_TOKEN -e commercial
sec run some-bot -e max -- just start
sec ls some-bot · sec diff some-bot -e commercial max

Объявление в .sec

Файл в папке сервиса декларирует инстансы и дефолт — читают и CLI (подставляет дефолт без -e), и агент. sec check [--all-envs] валидирует ключи инстанса(ов).

project: some-bot envs: commercial, max default: commercial keys: BOT_TOKEN, DB_PASSWORD

Ссылки и наследование общее значение — один источник

link — одно значение

Когда значение нужно нескольким пачкам — не копируй, а свяжи: proj/KEY отдаёт живое значение родителя, правка в родителе видна везде сразу.

sec link app/DB_URL shared/DATABASE_URL
get/run/export/render берут значение родителя
sec unlink app/DB_URL
сделать значение собственным (материализовать)

extend — вся пачка

Проект видит все ключи родителя read-only, собственные перекрывают; run/export инъектят объединение. В sec ls ссылки помечены →, унаследованные ⤷.

sec extend app --from base
· sec extend app --remove base
sec set app/KEY --override
дать проекту своё значение поверх унаследованного

Значение по ссылке/наследованию нельзя менять в потомке — set/gen/undo/import откажутся и подскажут родителя. Циклы отклоняются при создании; связи переживают backup/restore и sync. rm ключа/проекта, на который ссылаются или от которого наследуют, предупреждает (что станет битым, кто потеряет ключи), но выполняется — то же для get --once.

Организация и проверка

ls / mv / cp / rm

sec ls [proj] [-l|--json] [-e env]
сервисы → инстансы → ключи (без значений)
sec mv whois/OLD whois/NEW
перенос/переименование без раскрытия
sec cp whois/KEY staging/KEY
копия, оригинал на месте
sec rm whois/KEY · sec rm whois --all

check — манифест .sec

sec check [proj] [-e env | --all-envs]
все ли требуемые ключи заведены

Файл .sec — project/envs/default/keys (или просто список ключей по строке).

Перенос, синхронизация, ключ

backup / restore / sync

sec backup --file bak.enc
весь стор под passphrase (Argon2id + XChaCha20)
sec restore --file bak.enc [--replace]
merge (умолч.) или замена
sec sync --file ~/Dropbox/sec.enc
merge из блоба + push обратно; Keychain не нужен

rekey — ротация мастер‑ключа

sec rekey
новый ключ + перешифровка стора, с откатом при сбое

Значения сохраняются. Для SEC_KEY (env) новый ключ печатается — сразу прописать в окружение.

Мост с Infisical

Двусторонний обмен с Infisical (командный источник истины) через их CLI — нужен infisical login. Значения не текут в argv: pull берёт их из stdout экспорта, push — через временный .env 0600.

sec import whois --from-infisical --infisical-env dev
Infisical → локальный стор
sec push whois --to-infisical --infisical-env dev
локальный стор → Infisical

Инстанс в sec задаётся -e, окружение Infisical — отдельным --infisical-env (по умолчанию = значению -e). Без --projectId проект берётся из окружающего .infisical.json. Бэкенд вынесен в пакет internal/infisical и заменяем (CLI сейчас, REST API — позже).

Как хранится и что защищает

Хранение

  • Стор — один файл ~/.local/share/sec/store.enc, целиком XChaCha20‑Poly1305 (192‑битный nonce), 0600, запись атомарная под flock.
  • Мастер‑ключ (32 байта) по порядку: env SEC_KEY → системное хранилище ОС → файл ~/.config/sec/key.
  • Системное хранилище — macOS Keychain (security) или Linux libsecret (secret-tool); ключ уходит утилите через stdin — в ps не светится. Нет хранилища → файл / SEC_KEY.

Модель угроз

Защищает от случайной утечки секретов в чаты агентов, историю shell, скриншоты и логи. Не защищает от локального root/малвари — как любой менеджер с расшифровкой на той же машине.

Буфер: то, что sec кладёт сам (--clip), на macOS помечается ConcealedType и не пишется в историю менеджеров (Raycast, Maccy…). Значение, скопированное пользователем самим (set --clipboard), могло уже осесть в истории.

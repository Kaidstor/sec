# sec

Локальный менеджер секретов для проектов, сделанный под безопасную работу с
агентами (Claude Code и др.): значения секретов **никогда не появляются в
argv, истории shell и чате агента**. Ввод — скрытый / из stdin / из буфера
обмена; потребление — через env-инъекцию (`run`) или запись в файл (`export`).

Эти исходники лежат в том же репозитории, что и скилл `sec` (`../SKILL.md`) —
но в установленный через `npx skills` скилл они не входят, там только `SKILL.md`.
Единственная зависимость — `golang.org/x/crypto` (Argon2id для бэкапов) —
завендорена (`vendor/`), так что сборка из исходников офлайн, только через Go.
Но **ставить CLI обычно надо без Go** — готовым бинарём через Homebrew (см. ниже).

## Установка и релизы

### Готовый бинарь (без Go) — Homebrew

```sh
brew install kaidstor/tap/sec   # ставит собранный бинарь, Go не нужен
sec version
```

Формула живёт в tap-репозитории `kaidstor/homebrew-tap` и обновляется
автоматически на каждый релиз (GoReleaser).

### Из исходников (нужен Go)

Для разработки самого CLI. Из папки `cli/`:

```sh
just build            # → cli/sec (version=dev)
just build-versioned  # то же, но версия из git-тега
just install          # go install в ~/go/bin → прямой вызов `sec`
```

### Как выпустить новую версию

Версия проставляется линковщиком (`-ldflags -X …command.version`); релизные
бинарники и Homebrew-формулу собирает GoReleaser (`../.goreleaser.yaml`).

Разовая настройка (**✅ для этого репозитория уже выполнена** — шаги ниже
оставлены как справка для форка / повторения на новом репо):

1. Запушить этот репозиторий на GitHub как `github.com/kaidstor/sec`
   (публичный — Homebrew качает ассеты релиза). Модуль лежит в `cli/`; если
   делаешь отдельный CLI-репо с `go.mod` в корне — в `.goreleaser.yaml` поменяй
   `dir: cli` → `dir: .`.
2. Создать публичный tap-репо `github.com/kaidstor/homebrew-tap`.
3. Добавить в secrets репо `HOMEBREW_TAP_GITHUB_TOKEN` — PAT с правом записи в
   tap-репо (обычный `GITHUB_TOKEN` не может пушить в чужой репозиторий).
4. Лицензия — MIT (`../LICENSE`), в `.goreleaser.yaml` указана та же.

Каждый релиз — одной командой из корня репо:

```sh
./release.sh          # patch-бамп: следующий тег vX.Y.(Z+1) → пуш → релиз
./release.sh minor    # или minor / major / явная vX.Y.Z
./release.sh -n minor # dry-run: показать версию и проверки без тега
```

`release.sh` проверит ветку `main`, чистоту дерева и синхрон с `origin`, прогонит
`gofmt/vet/test`, поставит и запушит тег (это и запускает
`.github/workflows/release.yml`), а с установленным `gh` — дождётся зелёного
прогона. Вручную то же: `git tag v0.1.1 && git push origin v0.1.1`. Локальная
проверка конфига без публикации — `just release-check` / `just release-snapshot`
(бинарники в `dist/`).

После этого `brew install kaidstor/tap/sec` ставит свежую версию.
Проверить локально версию сборки: `sec version`.

Первый релиз этого репозитория — `v0.1.0`; каждое следующее обновление
выпускается новым тегом `vX.Y.Z`.

## Команды

```sh
sec set <proj>/<KEY>                 # сохранить: скрытый ввод (дважды)
sec set <proj>/<KEY> --clipboard     # взять значение из буфера обмена
sec set <proj>/<KEY> --clipboard --clear   # …и очистить буфер после
cat token.txt | sec set <proj>/<KEY> # из stdin / файла
sec gen <proj>/<KEY> [--len 32] [--symbols] [--clip]  # сгенерировать, не показывая
sec get <proj>/<KEY>                 # напечатать значение
sec get <proj>/<KEY> --clip [--clear-after 20s]  # в буфер (+авто-очистка)
sec get <proj>/<KEY> --peek          # маска ab…yz + длина (безопасно для чата)
sec get <proj>/<KEY> --fingerprint   # отпечаток fp:… (безопасно для чата)
sec get <proj>/<KEY> --prev 1        # предыдущее значение (история до 5 версий)
sec get <proj>/<KEY> --once          # показать и сразу удалить (одноразовая передача)
sec verify <proj>/<KEY>              # сверить кандидата с сохранённым (да/нет)
sec history <proj>/<KEY> [--json]    # версии: маска ab…yz, длина, дата (+ redo)
sec undo <proj>/<KEY>                # шаг назад по истории
sec redo <proj>/<KEY>                # вернуть значение, отменённое undo
sec forget <proj>/<KEY>              # стереть историю и redo (после ротации)
sec meta <proj>/<KEY> [--note … --kind … --rotate-every 90d --expires …]  # метаданные
sec otp <proj>/<KEY> [--clip]        # TOTP-код из seed (base32 / otpauth://)
sec log [proj[/KEY]] [-n 20] [--json]  # журнал обращений: кто/когда/что
sec ls [--json]                      # список проектов (без значений)
sec ls <proj> [-l|--json]            # ключи проекта (+даты, история, мета с -l)
sec ls [proj] --filter <шаблон>      # только совпавшие имена (подстрока или glob)
sec find <шаблон> [-l|--json]        # найти ключи по всему хранилищу → proj/KEY
sec diff <projA> <projB>             # сравнить проекты по отпечаткам (без значений)
sec mv <proj>/<KEY> <p2>[/<KEY2>] [--force]  # перенести/переименовать без раскрытия
sec cp <proj>/<KEY> <p2>[/<KEY2>] [--force]  # скопировать ключ (оригинал на месте)
sec link <proj>/<KEY> <род>/<KEY2> [--parent-env e]  # живая ссылка на чужое значение
sec unlink <proj>/<KEY> [--drop]     # снять ссылку (материализовать значение / удалить)
sec extend <proj> --from <род> [--parent-env e]  # наследовать ключи пачки read-only
sec extend <proj> [--remove <род>]   # без флагов — показать родителей; --remove — отвязать
sec set <proj>/<KEY> --override      # перебить ссылку/наследование своим значением
sec rm <proj>/<KEY>                  # удалить ключ
sec rm <proj> --all                  # удалить проект целиком
sec run <proj> -- cmd args...        # запустить cmd с env из проекта
sec run <proj> --only A,B -- cmd     # инъектить только выбранные ключи
sec export <proj> --file .env        # записать .env (файл 0600)
sec import [proj] [--file .env]      # импортировать ключи из .env
sec import [proj] path/to/.env       # путь можно и позиционно (проект — из cwd)
sec import <proj> --from-infisical   # импортировать из Infisical (их CLI)
sec push <proj> --to-infisical       # отправить ключи проекта в Infisical
sec check [proj] [-e env|--all-envs] # проверить наличие ключей из .sec-манифеста
sec scan <path...|-|--staged>        # найти сохранённые значения в файлах/диффе
sec redact <path...|->                # вычистить сохранённые значения из текста → stdout
sec render <tpl> --file <out> [--proj p]  # шаблон {{ secret "proj/KEY" }} → файл
sec stale [proj] [--older-than 90d]  # какие секреты пора ротировать
sec doctor                           # здоровье стора: права, дубли, ротация
sec backup --file <f>                # весь стор одним файлом под passphrase
sec restore --file <f> [--replace]   # восстановить: merge (умолч.) или замена
sec sync --file <blob>               # синхронизация через общий passphrase-блоб
sec rekey                            # ротация мастер-ключа с перешифровкой стора
sec info [--json]                    # путь, бэкенд ключа, статистика
sec completion zsh|bash|fish         # скрипт автодополнения для шелла
```

Если `<proj>` не указан (просто `KEY` в set/get/rm, или совсем опущен в
run/export/import) — берётся **имя текущей директории**. Внутри
`~/Projects/rebrandy/whois` команда `sec run -- just dev` возьмёт
секреты проекта `whois`.

У сервиса с несколькими инстансами (компании/стенды) выбирай набор флагом
`-e/--env` — см. раздел «Инстансы / окружения» ниже.

## Поиск и фильтрация

Когда проектов и ключей много, листать весь стор незачем: `find` ищет по всему
хранилищу, `ls --filter` сужает конкретный список. Шаблон в обоих —
**подстрока без учёта регистра** либо **glob** (`*` — любая последовательность,
`?` — один символ), если в нём есть `*`/`?`. Фильтр видит только имена
(проекты, инстансы, ключи) — значения секретов через него не проходят.

```sh
sec find token             # где угодно: имя проекта или ключа содержит "token"
sec find '*_URL' -l        # glob + даты обновления и метаданные
sec find 'gidcaf/*TOKEN'   # со слэшем: левая часть — проект, правая — ключ
sec find db --json         # машинно: ref/project/env/key/updatedAt/meta
sec find BOT_TOKEN -e max  # только инстанс max

sec ls --filter git        # список проектов, только совпавшие имена
sec ls gidcaf -f TOKEN     # ключи проекта, только совпавшие имена
```

`find` печатает адреса `proj/KEY` (у инстанса — `proj/KEY -e env`), ключи-ссылки
помечены `→ <родитель>`. Ищет по собственным ключам проектов — унаследованные
(`extend`) найдутся у родителя, где и живёт значение. Если ничего не нашлось,
код возврата 1 (как у grep) — удобно как условие в скрипте. У `ls --filter`
пустой результат — не ошибка (код 0).

## Импорт и экспорт `.env`

Путь к файлу можно передать позиционно или флагом `--file` (умолчание — `./.env`),
проект — аргументом либо по имени текущей папки:

```sh
sec import path/to/.env           # проект = имя текущей папки
sec import whois path/to/.env     # явный проект
sec import whois --file .env.prod # то же флагом
sec export whois --file .env      # обратно в файл (0600)
```

Диалект `.env` минимальный и предсказуемый:

- `#` в начале строки — комментарий; внутри значения — обычный символ
  (`PASS=p@ss#word!` сохранится целиком: инлайн-комментариев нет намеренно,
  иначе пароль с решёткой молча обрезался бы);
- префикс `export ` игнорируется;
- одинарные кавычки — литерал (`'\n$VAR'` как есть), двойные — с экранированием
  `\"`, `\\`, `\n`, `\r`; без кавычек значение обрезается по краям от пробелов
  (нужны краевые пробелы — бери в кавычки);
- BOM и CRLF от windows-редакторов не мешают;
- строки без `=` и с негодным именем переменной пропускаются с предупреждением
  в stderr (печатается имя, не значение).

`export` пишет тем же диалектом и сам заключает в кавычки всё, что содержит
спецсимволы или краевые пробелы, — поэтому `export → import` возвращает ровно
исходное значение (проверяется roundtrip-тестом).

## Автодополнение

`sec completion zsh|bash|fish` печатает скрипт дополнения для шелла. Установка
(zsh — добавить в `~/.zshrc` после `compinit`):

```sh
eval "$(sec completion zsh)"        # zsh
eval "$(sec completion bash)"       # bash (в ~/.bashrc)
sec completion fish > ~/.config/fish/completions/sec.fish   # fish
```

При установке через **Homebrew** completion ставится автоматически (goreleaser
кладёт скрипты в формулу) — ручной шаг нужен только для dev-сборки из исходников.

Дополняются подкоманды, флаги, а также **динамически из стора** — имена проектов,
ключи (`proj/<TAB>`) и инстансы (после `-e`). Скрипт дёргает скрытый бэкенд
`sec __complete`, который печатает только **имена** (значения секретов туда не
попадают). Это удобство для человека в терминале — агент вызывает `sec`
неинтерактивно и дополнением не пользуется.

## История значений

Каждый `set`/`import` поверх существующего ключа утаскивает старое значение в
историю (до 5 версий). `history` показывает версии маскированно (`ab…yz`,
длина, дата) — удобно свериться, не раскрывая значений; `get --prev N` достаёт
старое значение, не меняя текущего.

Навигация по истории — пара **undo/redo** с семантикой редактора: `undo`
шагает на одну версию назад (повторный — ещё глубже, до самой старой, потом
стена), `redo` возвращает вперёд; новый `set`/`gen` обнуляет redo-стек.
Спасает, когда токен перезаписали, а старый ещё нужен.

**Ротация скомпрометированного секрета:** обычный `set`/`gen` нового значения
не удаляет старое — оно остаётся в истории ещё до 5 перезаписей. `sec forget
<proj>/<KEY>` вычищает всю историю и redo, оставляя только текущее значение
(необратимо — делать после того, как новое значение проверено рабочим).

## Журнал обращений

Все обращения к секретам (`set`/`gen`/`get`/`run`/`export`/`import`/`rm`/`undo`/
`redo`/`forget`/`mv`/`cp`/`link`/`unlink`/`extend`/`meta`/`otp`/`verify`/`scan`/`sync`/`rekey`/`push`/…)
пишутся в `audit.jsonl` рядом с хранилищем: время, операция, ключ, детали
(например `env += TOKEN → just` или `показано prev=1`) и имя родительского
процесса (`zsh`, `node`, …). **Значения в журнал не попадают никогда.**
Смотреть: `sec log`, фильтр по проекту/ключу, `-n` — глубина. Ротация при
~1 МБ в `audit.jsonl.1`.

## Переносной бэкап

`sec backup --file bak.enc` пишет весь стор одним файлом, зашифрованным
passphrase (Argon2id, memory-hard, 64 МиБ / 3 прохода → XChaCha20-Poly1305) —
Keychain не нужен, файл можно переносить куда угодно. Формат `SECBAK03`: параметры KDF
лежат в заголовке блоба, поэтому их можно поднимать, не ломая старые файлы. `sec restore --file bak.enc` по
умолчанию делает **merge**: значения из бэкапа побеждают, локальные уходят в
историю (ничего не теряется); `--replace` заменяет стор целиком. Passphrase
спрашивается скрытым вводом, для скриптов/CI — env `SEC_PASSPHRASE`.

## Шаблоны конфигов

`sec render config.tpl --file config.yml` — Go text/template с функцией
`{{ secret "proj/KEY" }}` (или `{{ secret "KEY" }}` — проект из `--proj` либо
имени текущей папки). Для конфигов не-env формата: yaml, json, toml.
Как и export — пишет только в файл (0600), в stdout принципиально нет.

## TOTP (2FA)

Сохрани seed аутентификатора (base32-строка или целиком `otpauth://` URI —
digits/period/algorithm учитываются) обычным `set`, дальше `sec otp proj/KEY`
печатает одноразовый код и сколько секунд он ещё живёт. Код эфемерный, светить
его не страшно.

## Метаданные и ротация

У каждого ключа есть несекретные метаданные (назначение, где крутить, как
часто) — они живут отдельно от значения и переживают его перезапись.
`sec meta <ref> [--note … --kind … --rotate-url … --rotate-every 90d |
--expires 2026-12-01]`; note/kind можно задать сразу при `set`/`gen`. Без
флагов `sec meta <ref>` показывает метаданные (`--json` — машинно).

`sec stale [proj] [--older-than 90d]` перечисляет, что пора ротировать:
просроченное по политике (`expires` / `rotate-every` от даты обновления) либо
старше порога для ключей без политики. Ненулевой код удобен как гейт в CI.
`sec doctor` — общее здоровье стора: права файла (0600), бэкенд ключа,
доступность журнала, дубли значений (по отпечатку) и счётчик просроченных.

## Отпечатки и сверка

`sec get <ref> --fingerprint` печатает `fp:…` — HMAC-SHA256(мастер-ключ,
значение), усечённый до 64 бит. Ключуется мастер-ключом, поэтому безопасен для чата (по
отпечатку не перебрать значение) и сравним только внутри своих сторов с одним
ключом. На нём построены `sec diff <projA> <projB>` (какие ключи
совпадают/различаются, без раскрытия) и `--json`-вывод `history`/`ls`.
`sec verify <ref>` сверяет переданное значение (скрытый ввод / stdin / буфер)
с сохранённым и отвечает «совпадает/нет» — не раскрывая ни то ни другое.

## Скан на утечку

`sec scan` ищет сохранённые значения секретов в открытом виде — чтобы поймать
секрет до коммита/лога/публикации. `sec scan <path...>` (файлы и каталоги,
бинарники/.git/node_modules пропускаются), `sec scan -` (stdin), `sec scan
--staged` (`git diff --cached` — годится как pre-commit хук). Печатает только
`файл:строка → ключ`, значение не раскрывает; ненулевой код при находках.
Значения короче `--min` (8) игнорируются как шум; `--history` ищет и прошлые
значения.

## Редакция текста (redact)

`sec redact` — обратная сторона `scan`: не просто находит утечку, а отдаёт
**очищенный текст**. Каждое встреченное сохранённое значение заменяется на
`[redacted:proj/KEY]` (имя ключа безопасно для чата), остальной текст проходит
как есть. Вывод по построению безопасен — значений секретов в нём не остаётся,
поэтому его можно показывать в чате/логе.

```sh
just dev 2>&1 | sec redact          # прогнать вывод команды перед показом
sec redact app.log other.log        # очистить файлы → stdout
sec redact app.log --file safe.log  # записать результат в файл 0600
sec redact --mask <in               # заменять на глухое [redacted] без имени ключа
```

Флаги те же, что у `scan`: `--min` (умолч. 8, короче — шум), `--history` (чистить
и прошлые значения из истории — под `proj/KEY~prev`). Сводка «скрыто ключей: N
(…)» идёт в stderr, чтобы не мешать пайпу; в stdout — только очищенный текст.
Как и `scan`, redact знает только значения из стора: секрет, которого в sec нет,
он не увидит (для неизвестных секретов нужен паттерн-скан, а не redact).

## Инстансы / окружения (`-e/--env`)

У сервиса может быть несколько наборов значений одних и тех же ключей — разные
компании/стенды (`commercial`, `max`, `prod`, …). Это необязательная третья ось:
`<proj>` = сервис, `-e <инстанс>` выбирает набор. Внутри стор это отдельный
проект `service@env`, так что сервисы без инстансов работают как раньше.

```sh
sec set some-bot/BOT_TOKEN -e commercial   # значение инстанса commercial
sec get some-bot/BOT_TOKEN -e max          # значение инстанса max
sec run some-bot -e commercial -- just start
sec ls                                     # some-bot покажет: инстансы: commercial, max
sec ls some-bot                            # список инстансов с числом ключей
sec ls some-bot -e commercial              # ключи инстанса
sec diff some-bot -e commercial max        # сравнить инстансы по отпечаткам
```

Флаг `-e` действует почти во всех командах над значениями
(`set/gen/get/run/export/import/push/rm/mv/cp/meta/verify/otp/history/render/…`).
У `import`/`push` `-e` задаёт инстанс в sec, а окружение Infisical — отдельный
`--infisical-env` (по умолчанию берётся из `-e`).

## Ссылки и наследование (общие значения между пачками)

Когда одно значение нужно нескольким проектам (или инстансам) — не копируй его,
а **свяжи**: источник правды остаётся один, правки в родителе видны везде сразу.

- **`sec link <proj>/<KEY> <род>/<KEY2>`** — живая ссылка одного значения. `proj/KEY`
  своего значения не хранит; `get/run/export/render/otp/verify` берут текущее
  значение родителя. Инстанс родителя — `--parent-env` (умолч. как у ключа).
- **`sec extend <proj> --from <род>`** — наследование всей пачки: `proj` видит все
  ключи родителя read-only, собственные ключи перекрывают унаследованные. `run` и
  `export` инъектят объединение (свои + унаследованные). Показать/снять —
  `sec extend <proj>` / `--remove <род>`.

```sh
sec link app/DB_URL shared/DATABASE_URL    # app/DB_URL = живое значение родителя
sec extend app --from base                 # app видит все ключи base read-only
sec ls app                                 # ключи-ссылки помечены →, унаследованные ⤷
```

Значение по ссылке/наследованию **нельзя редактировать в потомке** — `set`/`gen`/
`undo`/`redo`/`forget` откажутся и подскажут родителя; `import` такие ключи
пропускает. Явные выходы: `sec unlink <proj>/<KEY>` (сделать значение
собственным — материализует текущее значение родителя, `--drop` просто удаляет
ключ) и `sec set <proj>/<KEY> --override` (дать проекту своё значение поверх
унаследованного). Циклы (ссылок и наследования) отклоняются при создании; связи
переживают `backup/restore` и `sync`. `rm` ключа/проекта, на который ссылаются
или от которого наследуют, печатает предупреждение (какие ссылки станут битыми /
кто потеряет унаследованные ключи), но выполняется — то же для `get --once`.

## Манифест проекта (`.sec`)

Файл `.sec` в папке сервиса объявляет требуемые ключи и — опционально — инстансы:

```
# .sec
project: some-bot
envs:    commercial, max
default: commercial
keys:    BOT_TOKEN, DB_PASSWORD, COMPANY_ID
```

`default` подставляется, когда `-e` не указан (в папке `sec run -- just start`
возьмёт `commercial`). `sec check` проверяет, что все `keys` заведены в выбранном
инстансе (или во всех — `sec check --all-envs`), и возвращает ненулевой код при
нехватке — удобно при онбординге и как гейт. Старый формат-список (`# коммент` +
голые ключи по строке) по-прежнему понимается — это тот же `keys`.

## Синхронизация

`sec sync --file <blob>` подтягивает секреты из общего passphrase-блоба
(merge, локальные вытесненные значения уходят в историю), затем пишет сводный
стор обратно — обе стороны сходятся к объединению. Блоб — тот же формат, что и
`backup` (Argon2id + XChaCha20), мастер-ключ Keychain для него не нужен, поэтому его
можно держать в Dropbox/iCloud/на сетевом диске или пушить git-обёрткой.

## Мост с Infisical

Для команд, где секреты живут в Infisical (командный источник истины), есть
двусторонний обмен через их CLI (`infisical`, нужен `infisical login`):

- `sec import <proj> [-e инстанс] --from-infisical [--infisical-env dev --path / --projectId <id> --token <t>]`
  — тянет секреты (`infisical export --format=json`, значения из stdout) в
  локальный проект `<proj>`; существующие уходят в историю.
- `sec push <proj> [-e инстанс] --to-infisical [--infisical-env … --path … --only A,B]` — отправляет
  ключи проекта в Infisical (`infisical secrets set --file`, значения идут
  временным `.env` 0600, не через argv; файл удаляется сразу).

Без `--projectId` проект берётся из окружающего `.infisical.json` (как
`infisical run/export` в папке сервиса). Роль sec — локальный агент-безопасный
черновик рядом с Infisical, а не замена ему. Бэкенд вынесен в пакет
`internal/infisical` и заменяем: сейчас это CLI, позже без изменения команд
можно подключить REST API.

## Ротация мастер-ключа

`sec rekey` генерирует новый мастер-ключ и перешифровывает хранилище (значения
сохраняются). Порядок с откатом: сначала обновляется бэкенд ключа
(Keychain/файл), затем перешифровывается стор; при сбое ключ в бэкенде
возвращается на прежний, чтобы стор остался читаемым. Для `SEC_KEY` (env)
новый ключ печатается — его нужно тут же прописать в окружение.

## Как хранится

- **Хранилище** — один файл `~/.local/share/sec/store.enc`
  (override: `SEC_STORE`), целиком зашифрован XChaCha20-Poly1305
  (192-битный nonce — случайный nonce безопасен без учёта числа записей),
  права 0600, запись атомарная (tmp + rename) под файловой
  блокировкой (flock) — параллельные записи не теряются.
- **Мастер-ключ** (32 байта) ищется по порядку:
  1. env `SEC_KEY` (hex64) — для серверов/CI;
  2. **системное хранилище ОС** (`service=sec, account=master`) — дефолт на
     десктопе: macOS Keychain (`/usr/bin/security`) или, на Linux, Secret
     Service / GNOME Keyring через `secret-tool` (libsecret), если он
     установлен и поднят;
  3. файл `~/.config/sec/key` (0600, override: `SEC_KEY_FILE`).

  Создаётся автоматически при первом `set`/`import`. Бэкенд системного
  хранилища выбирается по ОС (`keyring_darwin.go` / `keyring_linux.go` /
  заглушка `keyring_other.go` для прочих) — на macOS ACL висит на `security`,
  а не на нашем бинаре, так что пересборка CLI не вызывает повторных запросов
  доступа. Ключ при записи передаётся утилите через stdin (`security -i` /
  `secret-tool store`) и в `ps` не светится. Если системного хранилища нет
  (headless Linux без демона, другие ОС) — падаем на файл/`SEC_KEY`.
- **Бэкап** = файл хранилища + мастер-ключ из системного хранилища:
  macOS — `security find-generic-password -s sec -a master -w`, Linux —
  `secret-tool lookup service sec account master`. Перенос на другую машину:
  скопировать store.enc и положить ключ любым из трёх способов. Между машинами
  удобнее `sec sync` — блоб шифруется passphrase'ой и от мастер-ключа не
  зависит, так что mac↔Linux синхронизируются без переноса ключа.

## Модель угроз

Защищает от **случайной утечки** секретов в чаты агентов, историю shell,
скриншоты и логи. Не защищает от локального root/малвари (как и любой
менеджер с расшифровкой на той же машине).

Буфер обмена: всё, что sec кладёт туда сам (`--clip`, `gen --clip`,
`otp --clip`), на macOS помечается `org.nspasteboard.ConcealedType`
(конвенция nspasteboard.org) — менеджеры истории буфера (Raycast, Maccy,
Alfred, Paste) такие записи в историю не пишут. Но значение, которое
пользователь скопировал **сам** (сценарий `set --clipboard`), исходное
приложение пометить не могло — оно уже осело в истории менеджера;
`--clear` чистит только текущий буфер, не историю.

## Ограничения

- `export`/`render` пишут плейнтекстовый .env на диск (0600) — это осознанный выход
  из шифрованного мира; файл не коммитить.
- Значения с `$` внутри при экспорте не экранируются под shell-source —
  dotenv-парсеры (bun, docker compose) читают их как есть, а вот
  `source .env` в bash может подставить переменную.
